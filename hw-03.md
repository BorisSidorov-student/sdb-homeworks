### Решение 1
Для решения задачи по развертыванию сервиса **`Elasticsearch`** буду использовать **`docker`**, а именно буду поднимать контейнеры через **`docker compose`**. Первым делом создам файл **`compose.yml`** в котором опишу развертывание сервиса **`elasticsearch`**. Из важного

### Решение 3
Продолжаю работать в проекте **`docker compose`**. В задании сказано запустить сервис **`Logstash`** и **`Nginx`**. В **`compose.yml`** я допишу конфигурацию развертывания сервиса **`Logstash`**, а **`Nginx`** у меня уже установлен с прошлых заданий и лог файл доступа уже имеется. В **`yml`** файле я дописал следующий блок по сервису **`Logstash`**:

- **`image: logstash:8.14.3`**
- **`container_name: logstash`**
- **`user: 1000:4`** — тут важная строка которая добавляет пользователю **`logstash`** с **`id 1000`** принадлежность к группе **`id 4`** (**`adm`**). Это решает проблему с доступом на чтение лог файла, ведь именно группе **`adm`** выставлен доступ на чтение в атрибутах на хосте.
- **`environment`**: добавлены две переменные.  
  **`PACK_MONITORING_ENABLED: "false"`** отключение **`XPACK мониторинга`**  
  **`ES_HOST: "elasticsearch:9200"`** адрес для **`elasticsearch`**
- **`ports: "5044:5044"`** пробрасываю порт
- **`volumes`**: тут я пробрасываю ряд томов в контейнер  
  - **`./configs/logstash/pipelines.yml:/usr/share/logstash/config/pipelines.yml`** — конфигурационный файл **`pipelines.yml`**  
  - **`./configs/logstash/pipelines:/usr/share/logstash/config/pipelines`** — целиком всю директорию **`pipelines`** в котором будут хранится все пайплайны для работы сервиса  
  - **`/var/log/nginx:/var/log/nginx/:ro`** — директория в которой хранится лог файл **`nginx`**, цель задания.
- **`depends_on: elasticsearch`** — зависимость от сервиса **`elasticsearch`**

Итоговый файл **`docker-compose .yml`** получился таким:  
[**`docker-compose-task3.yml`**](docker-compose/docker-compose-task3.yml)

В конфигурационном файле **`pipelines.yml`** из важного является строка с путем до директории с пайплайнами **`path.config: "/usr/share/logstash/config/pipelines/nginx.conf"`** и **`id`** пайплайна.  
[**`pipelines.yml`**](configs/logstash/pipelines.yml)

В самом **`pipeline`** для **`logstash`** на ввод описан плагин **`file`**:
```
input {
  file {
    path => "/var/log/nginx/example-http.com-acess.log"
    start_position => "beginning"
  }
}
```

В **`filter`** заранее проанализировав логи которые будут поступать, попросил нейронку сформировать регулярку для **`grok`**:
```
filter {
  grok {
    match => { "message" => "%{IPORHOST:client_ip} - - \[%{HTTPDATE:timestamp}\] \"%{WORD:method} %{URIPATHPARAM:request} HTTP/%{NUMBER:http_version}\" %{NUMBER:response_code:int} .* \"%{GREEDYDATA:user_agent}\"" }
  }
}
```

В вывод добавил стандартный вывод для быстрого просмотра и соответственно сам сервис **`elasticsearch`**, помимо хоста добавил и **`index`** с добавлением даты для удобства последующего анализа логов:
```
output {
  stdout {
  }
  elasticsearch {
      hosts => [ "${ES_HOST}" ]
      index => "logs_app_gen%-%{+YYYY.MM.dd}"
  } 
} 
```
[**`nginx.conf`**](configs/logstash/pipelines/nginx.conf)

### Задание 4. Filebeat. 

Установите и запустите Filebeat. Переключите поставку логов Nginx с Logstash на Filebeat. 

*Приведите скриншот интерфейса Kibana, на котором видны логи Nginx, которые были отправлены через Filebeat.*

---

### Решение 4
В **`инфраструктуре`** лучшей практикой является доставка логов в **`elasticsearch`** является **`logstash`**, но сам по себе этот сервис не рационально использовать в качестве агента на хосте, так как он много потребляет ресурсов хоста. Поэтому в качестве агента будет выступать сервис **`Filebeat`** который прекрасно работает с чтением файлов и передачи их содержимых уже либо напрямую в **`elasticsearch`** или передать эти данные для последующей обработки в **`logstash`**, чтобы в свою очередь был проведен парсинг полученных данных и потом уже готовые данные поступали в **`elasticsearch`** для удобного анализа через **`web`** сервис **`kibana`**. Эту модель я и буду реализовывать:  
**`filebeat > logstash > elasticsearch > kibana`**

Продолжаю работать с проектом **`docker compose`**. Добавляю сервис **`filebeat`**, важные директивы которые я описал:

- **`user: 1000:4`** — решает проблему со чтением логов **`nginx`**, где недостаточно прав на чтение лог файла **`nginx`** с хоста.
- **`volumes`**:  
  - **`./configs/filebeat/data/filebeat:/usr/share/filebeat/data`** — тут на самом деле выглядит как костыль, так как директива **`user`** в котором я изменил атрибуты возникает проблема с запуском контейнера так как возникает проблема с созданием мета файлов. Я создал внутри проекта необходимые директории и выставил владельца которые потребуются для записи в них метаданные, а через **`volume`** просто пробросил эту директорию внутрь контейнера. Давать рута в контейнера не стал так как это не лучшая практика, также не стал создавать именованный том.  
  - **`./configs/filebeat/filebeat.yml:/usr/share/filebeat/filebeat.yml`** — пробрасываю конфигурационный файл в котором будет описан процесс чтения лог файла и передачи на сервис **`logstash`**  
  - **`/var/log/nginx:/var/log/nginx/`** — пробрасываю директорию с лог файлом **`nginx`**
- **`environment`**:  
  **`LOGSTASH_URL: "logstash:5044"`** — адрес до сервиса **`logstash`**

Также в проекте я закомментировал строку на проброс директории с логами в сервисе **`logstash`**, так как теперь сбором логов будет выступать **`filebeat`**.  
`# - /var/log/nginx:/var/log/nginx/`

Так как я буду создавать новый пайплайн для работы **`logstash`** я добавил новую переменную в которой отмечу название того конфигурационного файла который буду использовать в настройках **`pipelines.yml`**  
**`ACTIVE_PIPELINE: filebeat`**

Итоговый файл **`docker compose yml`** выглядит так:  
[**`docker-compose-task4.yml`**](docker-compose/docker-compose-task4.yml)

Видоизмененный файл **`pipelines.yml`** в котором проставляется переменная из **`docker yml`**  
[**`pipelines-task4.yml`**](configs/logstash/pipelines-task4.yml)

В конфигурационном файле для сервиса **`filebeat`** описал директиву **`type: log`** и путь до целевого логфайла данные которого необходимо будет передавать в сервис **`logstash`**. Так же добавил ещё одно поле **`acess.log: nginx`**. Которое появится в **`kibana`** для анализа. Файл получился таким:  
[**`filebeat.yml`**](configs/filebeat/filebeat.yml)

Создал новый **`pipeline`** для **`logstash`** в котором изменил блок **`input`**, указав плагин **`beats`** и порт **`5044`**. Этой конфигурацией говорю **`logstash`** принимать данные от агентов семейства **`Beats`**, куда соответственно сервис **`filebeat`** и входит. Все остальное не менял. Итоговый файл получился таким:  
[**`filebeat.conf`**](configs/logstash/pipelines/filebeat.conf)
